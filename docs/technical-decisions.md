# RingCraftLab リフィル作成ツール 技術選定メモ

## 1. 目的

本ドキュメントは、RingCraftLab リフィル作成ツールにおける主要な技術選定と、その理由を整理するためのものである。

本アプリは、スマートフォンやタブレット内の画像を、システム手帳用リフィルとして印刷可能なPDFへ変換するツールである。  
そのため、一般的な画像表示UIだけでなく、面付け、原寸配置、穴位置、カット線、印刷時のズレ対策など、印刷物としての要件を満たす必要がある。

## 2. 技術スタック

想定技術スタックは以下とする。

- React
- TypeScript
- Vite
- Tailwind CSS
- Canvas API
- pdf-lib
- Vitest
- Playwright

## 3. フロントエンド基盤

### 3.1 React

UI構築の中心としてReactを採用する。  
画面遷移、状態管理、画像一覧、プレビューUI、編集UIなどをコンポーネント単位で整理しやすい。

### 3.2 TypeScript

寸法計算、面付けロジック、穴パターン定義、PDF配置データなど、型で厳密に扱いたい情報が多いためTypeScriptを採用する。  
特に本ツールでは、mm単位の寸法と画面上の表示座標を区別して扱う必要があるため、構造化された型定義が重要である。

### 3.3 Vite

本アプリはクライアントサイド中心の構成であり、サーバーサイドレンダリングを前提としない。  
Viteは起動とビルドが軽く、React + TypeScript の構成を素直に組めるため採用する。

### 3.4 Tailwind CSS

本ツールはコンビニコピー機風の独自UIを目指しており、既存デザインシステムへ寄せるよりも、画面ごとに意図した見た目を組み立てやすいことが重要である。  
Tailwind CSS は細かいUI調整とプロトタイピングの両立がしやすく、本件に向いている。

## 4. PDF生成方式

### 4.1 採用方針

PDF生成には `pdf-lib` を採用する。  
プレビュー画面をキャプチャしてPDF化する方式ではなく、面付け計算の結果をもとにPDFを直接描画する方式を採用する。

### 4.2 採用理由

本アプリでは、以下の情報をmm単位で正確に扱う必要がある。

- リフィル外形
- 穴位置
- カット線
- 穴回避領域
- リフィルごとの画像配置
- A4上の面付け位置
- 複数ページ構成

これらは「画面の見た目」ではなく、「印刷物の寸法情報」である。  
そのため、画面を画像としてPDFへ貼る方式より、座標を直接指定してPDFを組み立てる方式の方が適している。

## 5. プレビュー方式

### 5.1 採用方針

プレビューには `Canvas API` を使用する。

### 5.2 役割

Canvas API は、PDF生成の代替ではなく、**同じ配置データをもとに画面上のプレビューを描くため** に使う。

構成は以下を想定する。

```txt
面付けロジック
  ↓
配置データ
  ↓                ↓
Canvasプレビュー    pdf-libによるPDF生成
```

この構成により、プレビューとPDFが別実装で完全に独立するのではなく、同じ設計データを共有できる。

## 6. html2canvas を採用しない理由

### 6.1 過去実装での方式

過去の実装では、以下の構成を採用していた。

- プレビューDOMを構築
- html2canvasでビットマップ化
- jsPDFで画像としてPDFに貼り付け

この方式は、プレビューと出力結果を一致させやすいという利点があった。

### 6.2 過去実装で発生した課題

一方で、以下の課題があった。

- プレビューとPDFの位置ズレ
- 画面pxとPDF上のmm変換による誤差
- 印刷時にサイズが狂いやすい
- html2canvas の描画結果に依存する
- 高解像度出力時の画質と処理コストの問題
- object-fit の扱いに補助処理が必要

特に本件では、以下のズレが重要な問題になる。

- リフィル寸法
- 穴位置
- カット線位置
- 画像の切れ方

### 6.3 今回不採用とする理由

今回は、画面キャプチャによるWYSIWYG性よりも、**印刷寸法の正確性と拡張性** を優先する。  
そのため、html2canvas方式は採用しない。

## 7. jsPDF を採用しない理由

jsPDF はクライアント側でPDFを作成する手段として有力だが、今回は `pdf-lib` を優先する。

理由は以下の通り。

- 面付け結果を直接PDF上へ描きたい
- 複数ページ構成を座標ベースで管理したい
- 画像、線、マーカーを同じ座標系で扱いたい
- 将来的な拡張を見据えて、PDF構造をより直接的に組み立てたい

## 8. クライアントサイド処理を基本とする理由

本アプリでは、ユーザーの画像をサーバーへ送信せず、ブラウザ内で処理する方針を採用する。

理由：

- 写真やスクリーンショットを外部へ送らずに済む
- 個人情報を含む可能性がある画像を扱いやすい
- 小規模構成で公開しやすい
- ホスティングコストを抑えやすい

## 9. テスト方針

### 9.1 Vitest

面付けロジック、穴位置計算、ページ数計算、向き自動決定など、純粋関数として検証しやすい部分には Vitest を使う。

主なテスト対象：

- サイズ定義
- 穴パターン計算
- 配置枚数計算
- ページ分割
- 余剰領域計算

### 9.2 Playwright

UI操作と出力前確認には Playwright を使う。

主な確認対象：

- モバイル幅での操作
- ステップ遷移
- 画像選択からプレビューまでの導線
- 複数ページ時の表示
- PDF生成ボタンまでの流れ

## 10. モバイルファースト設計

本アプリは、スマートフォンやタブレットで画像を選択し、そのままPDFを作る利用シーンを主対象とする。  
このため、UI設計はPC中心ではなく、モバイルファーストとする。

判断理由：

- 入力画像の多くがスマホ内の写真やスクリーンショットである
- コンビニ印刷や家庭印刷の前段として、移動中や手元で操作しやすい形が望ましい
- 大きなボタンと段階的な設定UIが適している

## 11. UI設計方針

操作フローは以下を基本とする。

1. 作り方を選ぶ
2. 手帳サイズを選ぶ
3. レイアウトを選ぶ
4. 画像を選ぶ
5. 編集する
6. PDFを作る

このフローは、コンビニコピー機の分かりやすさと、テンプレート選択型ツールの理解しやすさの両方を意識している。

## 12. 将来拡張を見据えた設計方針

初期リリースではMVPに絞るが、内部設計は将来の追加機能に耐えられるようにする。

想定拡張：

- A3、A5出力
- カスタムサイズ
- 余白配分切り替え
- 確認スケール
- 穴位置マークの円表示
- 並び替え
- 個別部数指定
- 両面印刷対応

そのため、以下は最初から分離して管理する。

- リフィルサイズ
- 穴パターン
- 配置データ
- 画像編集情報
- 面付けロジック
- PDF描画処理
- プレビュー描画処理

## 13. まとめ

今回の技術選定では、過去のDOMキャプチャ方式での課題を踏まえ、以下を優先した。

- 印刷寸法の正確性
- 面付けロジックの明確化
- プレビューとPDFの一貫性
- モバイル操作のしやすさ
- 将来拡張しやすい構造

そのため、React + TypeScript + Vite を基盤とし、プレビューに Canvas API、PDF生成に pdf-lib を採用する。
