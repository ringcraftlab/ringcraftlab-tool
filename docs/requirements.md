# RingCraft Refill Maker Requirements

## 1. Project goal

スマホ内の写真やスクリーンショットを、システム手帳に挟めるリフィルとしてPDF出力できるWebツールを作る。

就職用ポートフォリオとして、以下の2点を両立する。

- 実際に使えること
- スマホ画面でも価値がすぐ伝わること

## 2. Target users

- システム手帳を日常的に使っている人
- 学校連絡、習い事予定、地図、予約画面、写真、レシピ、メモを紙でも見返したい人
- 主にスマホやタブレットから操作する人

## 3. MVP scope

### Included

- 画像アップロード
- 手帳サイズ選択
- レイアウト選択
- 画像の個別調整
- A4 PDF出力
- 複数ページPDF出力
- 穴位置マーク表示
- カット線表示
- 左穴 / 右穴対応

### Not in MVP

- 両面印刷前提の高度な見開き設計
- 並び替え機能
- 任意穴径の細かな編集UI
- A3 / A5出力UI
- 会員登録

## 4. Supported refill sizes

- ミニ5: 61 x 105 mm
- ミニ5スクエア: 105 x 105 mm
- ミニ6: 80 x 126 mm
- バイブル: 95 x 170 mm
- A5: 148 x 210 mm

## 5. Output paper

### Initial release

- A4のみ

### Future expansion

- A3
- A5

内部設計上は「手帳サイズ」と「出力用紙サイズ」を分離して持ち、将来の追加に備える。

## 6. Core user flow

ツール画面は4ステップ構成。コンビニのネットプリント・インデックスプリントを参考にした操作感。

1. 画像を選ぶ（複数可・長押しドラッグで並び替え可能）
2. 手帳サイズを選ぶ
3. 作り方を選ぶ（複数の画像を並べる / 同じ画像を繰り返す）
4. 確認・PDF出力（A4面付けプレビュー・リフィルタップで個別編集・PDF出力）

個別編集画面で設定できるもの:

- ズーム・位置調整・90度回転・リセット
- 穴の位置（左穴 / 右穴）
- レイアウト（余白あり / 全面）

補足:

- 穴位置・レイアウトは画像ごとに個別設定できる
- Step 4の一覧画面がPDF出力ボタンも兼ねる

## 7. Layout and image placement options

### Image placement modes

- リングを避ける印刷
- 全面印刷

### Default marker visibility

- カット線: ON
- 穴位置マーク: ON

### Hole mark style

- 初期実装は `+`
- 将来的に円形穴表示へ差し替え可能な構造にする

### Hole avoidance width

- ミニ5 / ミニ6: 8 mm
- バイブル / A5 / ミニ5スクエア: 11 mm

## 8. Imposition policy

基本方針は「できるだけたくさん配置する」。

ただし、余白計算は将来的に以下へ拡張できる構造にする。

- 用紙いっぱいに配置
- 余白を均等配分して配置

MVPでは、自動面付けで配置数を決定し、余り領域は均等割りできる設計を前提にする。

## 9. Image editing requirements

MVPで必要な操作:

- 拡大 / 縮小
- 位置調整
- 90度回転
- リセット

不要:

- 自由角度回転
- フィルタ加工

実装済み:

- 並び替え（長押しドラッグによる順番変更）

画像順はアップロード順を初期値とし、ユーザーが並び替え可能とする。

## 10. PDF requirements

- 出力形式はPDF
- 複数ページは1つのPDFにまとめる
- PDFは原寸前提で生成する
- リフィル外形は実寸で作る

## 11. Print guidance requirements

### Required notice

印刷時は以下の注意を必ず出す。

- `100%`
- `実際のサイズ`
- `等倍`

のいずれかで印刷すること。

以下を選ぶとズレる可能性があることを案内する。

- 用紙に合わせる
- 印刷可能領域に合わせる
- 縮小して全体を印刷

### Edge clipping notice

プリンターやコンビニコピー機では、100%印刷やフチなし印刷でも外周がわずかに欠ける場合がある。

そのため、編集画面では以下を満たす。

- 端切れ注意の考え方を持つ
- 必要なら警告エリアをプレビュー表示できる
- 重要な文字や顔が端に来ないようユーザーがズーム / 位置調整できる

MVPでは、強い固定余白を設けて枚数を減らすより、注意表示と編集で吸収する方針とする。

## 12. Hole side policy

- 左穴 / 右穴の切り替えに対応する
- 両面印刷の完全対応はMVP外
- 左右穴は単面利用でも価値があるため採用する

## 13. LP / top page requirements

LPはスマホファーストで、長すぎない構成にする。

### LP role

- このサイトで何ができるかを一瞬で伝える
- 最終成果物が「A4 PDF」ではなく「切って穴をあけて手帳に挟めるもの」だと伝える
- 実ツールへの導線を作る

### LP structure

LPは原則4セクション。

1. ファーストビュー
2. こんなものを手帳に入れられます
3. かんたん4ステップ
4. 最後のCTA

### LP first view requirements

- 完成後の見た目を最優先で見せる
- スマホ内の情報が手帳に入った状態を大きく見せる
- CTAはすぐ見える位置に置く
- 長い説明文は置かない

### LP examples section

例として見せる内容:

- 学校のお知らせ
- 習い事の予定
- 地図 / アクセス
- 旅行の予約情報
- 家族の予定 / 写真
- レシピ / メモ

### LP step section

手順はLP上では簡略化してよい。

- 画像を選ぶ
- 手帳サイズ / レイアウトを選ぶ
- 調整する
- PDFを作って印刷する

### LP details policy

- サイズ詳細は小さなアイコンやチップ程度でよい
- 細かな設定はLPで見せすぎない
- 詳細設定は実ツール側の編集画面で扱う

## 14. Visual direction

- コンビニコピー機のような、わかりやすい操作感を参考にする
- ただし古い業務機の無骨さをそのまま再現するのではなく、スマホ向けに整理する
- 情報量は絞る
- 白ベースで、必要箇所に青と黄を効かせる方向は相性がよい
- 「印刷設定の説明サイト」ではなく、「使ってみたくなる道具」に見せる

## 15. Screen policy

### LP

- 価値訴求を優先
- 文章は短く
- スクロール量は短め

### Tool screens

- 操作優先
- 1画面1目的を徹底
- スマホ片手でも意味がわかるラベルにする

## 16. Quality bar

- 仕様として決めた内容を画面に反映する
- 見た目だけでなく、印刷用途として破綻しないことを重視する
- UI提案時は毎回、要件とのズレを先に確認する
